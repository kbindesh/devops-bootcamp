# Jenkins Pipeline

## What is a Jenkins Pipeline?

- Jenkins pipeline, in simple terms, is a group of multiple Jenkins jobs that run in sequence or in parallel or a combination of both.

- **Example**: Jenkins Continuous Delivery pipeline

  ```
  Continuous Delivery (CD) Pipeline

    - Jenkins Job-01 - Static code analysis

    - Jenkins Job-02 - Publish to Artifactory

    - Jenkins Job-03 - Deploy to Testing server

    - Jenkins Job-04 - User Acceptance Test

    - Jenkins Job-05 - Preformance Test
  ```

### 02. Sections

#### 2.1 agent

- **top-level agent**

  ```
  pipeline {
      agent any
      stages {
          stage('stage_label') {
              steps {
                  echo 'Hello there'
              }
          }
      }
  }
  ```

- **stage-level agent**

  ```
  pipeline {
      agent none
      stages {
          stage('stage_label') {
              agent any
              steps {
                  echo 'Hello World'
              }
          }
      }
  }
  ```

The agent section parameters are as follows:

1. **any**
   - Execute the Pipeline, or stage, on any available _agent_ (aka slave or node).
   - For example: `agent any`
2. **none**
   - Is applied at the top-level of the pipeline block.
   - No global agent will be allocated for the entire Pipeline run and each stage section will need to contain its own agent section.
   - For example: `agent none`
3. **label**
   - Execute the Pipeline, or stage, on an agent available in the Jenkins environment with the provided label.
   - For example: `agent { label 'my-defined-label' }`

4. **node**
5. **docker**
   - Execute the Pipeline, or stage, within the given container which will be dynamically created on a node.

   ```
   agent {
      docker {
          image 'maven:latest'
          args  '-v /tmp:/tmp'
          registryUrl 'https://myregistry.com/'
          registryCredentialsId 'myPredefinedCredentialsInJenkins'
      }
    }
   ```

6. **dockerfile**
7. **kubernetes**

#### 2.2 tools

- Defines the tools or build environment to be used in the pipeline.
- Supported Tools
  - maven
  - jdk
  - gradle

  ```
  pipeline {
      agent any
      tools {
          maven 'maven-3.9.11'      // The tool name must be defined in Jenkins under Manage Jenkins → Tools
      }

      stages {
          stage('Get Maven Version') {
              steps {
                  sh 'mvn --version'
              }
          }
      }
  }
  ```

#### 2.3 stages

- The stages section may contain one or more than one stage directives (sub-section).
- The stage section defines the real tasks to be carried out by the Jenkins pipeline.
- **Recommendation**
  - Have at least one stage directive for each discrete part of the continuous delivery (CD) process such as:
    1. Build
    2. Test
    3. Deploy

  ```
  pipeline {
      agent any
      stages {
          stage('Build App') {
              steps {
                  echo 'Building the App'
              }
          }
          stage('Unit Test App') {
              steps {
                  echo 'Test the App'
              }
          }
          stage('Deploy App') {
              steps {
                  echo 'Deploying the App'
              }
          }
      }
  }
  ```

#### 2.4 post

- The post section defines one or more additional steps that you want to run upon the successful completion of a jenkins pipelines or stage's run.
- You can add the _post_ section in the outer-most _pipeline_ block or inside a particular _stage_ block.
- **Conditions**: _always, changed, fixed, aborted, failure, success, failure, unsuccessful, unstable._

  ```
  pipeline {
      agent any
      stages {
          stage('stage_label') {
              steps {
                  echo 'Hello there'
              }
          }
      }
      post {
          always {            // Post condition
              echo 'I will always say Hello!'
          }
      }
  }
  ```

### 03. Directives

#### 3.1 environment

- Environment variables are key-value pairs that store configuration settings, credentials, and other dynamic values used during build and deployment processes.
- Environment variables can be defined at various levels, including:
  - system-wide (Jenkins global settings)
  - per-node
  - per-job
  - within pipeline scripts
- You can define custom environment variables using _Jenkins UI_, _env_ directive in pipelines, or _external configuration files_.

```
# Setting local environment variables in declarative pipelines

pipeline {
    agent any
    environment {       // pipline-level env variable; accessible to all stages
        HELLO = 'WORLD'
    }
    stages {
        stage('Morning') {
            environment {           // stage specific env var
                GREETING = 'Good morning!'
            }
            steps {
                echo env.HELLO
                echo env.GREETING
            }
        }
        stage('Evening') {
            steps {
                echo env.HELLO
                echo env.GREETING     // This is invalid as GREETING env is defined in another stage
            }
        }
    }
}

```

- Some of the `Jenkins built-in environment variables` are as follows:
  - BUILD_NUMBER
  - JOB_NAME
  - BUILD_URL
  - NODE_NAME
  - WORKSPACE

- **Storing sensitive information in environment variables**

```
# To use an existing jenkins credential in a Jenkins pipeline, you can use "credentials" directive within environment block

pipeline {
    agent any
    environment { // accessible to all stages
        HELLO = 'WORLD'
        MY_SECRET = credentials("MY_SECRET_MSG") // credentials directive | pass credential ID as param
    }
    stages {
        stage('Morning') {
            environment {
                GREETING = 'Good morning!'
            }
            steps {
                echo env.HELLO
                echo env.GREETING
                echo env.MY_SECRET      // prints secret text
            }
        }
        stage('Afternoon') {
            steps {
                echo env.HELLO
                echo env.GREETING       // This is invalid as GREETING env is defined in another stage
            }
        }
    }
}
```

- To view Jenkins environment variables list:
  - via _web browser_: `[Jenkins URL]/env-vars.html`
  - via shell command in a Jenkins job: steps { bat 'set' // for windows }

#### 3.2 parameters

- _parameter_ directive provides a list of parameters which you can define in the jenkinsfile.
- Each parameter has a Name and Value.
- Use _params_ object to access defined parameters in the pipeline.
- Parameter Types
  - string
  - text
  - booleanParam
  - choice
  - password

  ```
  pipeline {
      agent any
      parameters {
          string(name: 'PERSON', defaultValue: 'Mr Jenkins', description: 'Who should I say hello to?')

          text(name: 'BIOGRAPHY', defaultValue: '', description: 'Enter some information about the person')

          booleanParam(name: 'TOGGLE', defaultValue: true, description: 'Toggle this value')

          choice(name: 'CHOICE', choices: ['One', 'Two', 'Three'], description: 'Pick something')

          password(name: 'PASSWORD', defaultValue: 'SECRET', description: 'Enter a password')
      }
      stages {
          stage('Example') {
              steps {
                  echo "Hello ${params.PERSON}"

                  echo "Biography: ${params.BIOGRAPHY}"

                  echo "Toggle: ${params.TOGGLE}"

                  echo "Choice: ${params.CHOICE}"

                  echo "Password: ${params.PASSWORD}"
              }
          }
      }
  }
  ```

#### 3.3 input

- The _input_ directive on a stage allows you to prompt for getting input value from the user.
- The stage will pause after any options have been applied, and before entering the agent block for that stage or evaluating the when condition of the stage.
- If the input is approved, the stage will then continue.

- _Configuration options_: message, id, ok, submitter, parameters

  ```
  pipeline {
      agent any
      stages {
          stage('Example') {
              input {
                  message "Should we continue?"
                  ok "Yes, we should."
                  submitter "alice,bob"
                  parameters {
                      string(name: 'PERSON', defaultValue: 'Mr Jenkins', description: 'Who should I say hello to?')
                  }
              }
              steps {
                  echo "Hello, ${PERSON}, nice to meet you."
              }
          }
      }
  }
  ```

#### 3.4 when

- The _when_ directive allows the Pipeline to execute a stage conditionally.
- The when directive must contain at least one condition.

  ```
  pipeline {
      agent any
      stages {
          stage('Build App') {
              steps {
                  echo 'Hello World'
              }
          }
          stage('Deploy App') {
              when {
                  branch 'production'
              }
              steps {
                  echo 'Deploying'
              }
          }
      }
  }
  ```
